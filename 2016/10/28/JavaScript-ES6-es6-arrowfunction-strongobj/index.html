Arrow Function : 화살표 함수

ES6에 추가된 함수 관련된 표현/문법으로 화살표 함수라고 명명하며, 람다식의 영향을 받은 표현식이라고 한다. 이 같은 표현은 자바,파이썬,C# 등에서도 비슷하게 사용하는 용례를 볼 수 있는데 생각보다 역사가 오래된 표현식이라고 한다.(익명함수라고 표현할 수도 있다고 한다.)

특징

  • 간결한 표현 가능(function 키워드를 => 연산자로 대체)

  • 라인이 하나밖에 없는 함수의 경우 중괄호 {} 생략 가능

  • 인자 존재 유무 및 중괄호 라인 수에 따른 표현 방법의 다양

this

자바 같은 경우는 this 키워드가 가리키는 대상은 객체 자신이라 식별하기 쉽다. 하지만, 자바스크립트 같은 경우에는 그때그때 달라지는 this의 참조 때문에 골치를 썩는 경우가 많다.(기본 규칙은 존재하지만, 코드에서 this 바인딩을 바꿀 수 있기 때문에 라인이 긴 코드에서는 이를 식별하려면 디버깅하는 소요가 발생한다.)

화살표 함수의 경우, 기존 함수의 this 바인딩과 다른 규칙을 가지는데, 화살표 함수가 속한 스코프의 this를 참조 하는 규칙이 적용된다.

다시 말해, 함수 block을 감싸고 있는 외부 scope의 this를 참조한다는 이야기이다. 전역 코드에서 함수(화살표 함수)를 생성할 경우 외부 환경은 전역 스코프이며, 전역 스코프의 this는 window(브라우저 기준) 이기 때문에 화살표 함수 내부에서 this는 window를 가리키게 된다.

좀 더 응용할 수 있는 예제는 아래와 같다.

funcT 스코프에서 정의한 that 변수에 this(window를 참조하는)를 할당하고, 내부 함수(underFuncT) 내부에서 this와 that이 동일한 참조를 하고 있는 확인하고 있는 코드이다. 결과부터 말하자면 true이다. 왜냐하면, 화살표 함수를 감싸고 있는 외부의 스코프의 this를 참조하기 때문에 underFuncT의 외부 스코프는 funcT의 스코프이고 funcT의 스코프에서 참조하는 this는 window이기 때문이다.

다시 한번 정리하자면, 화살표 함수에서 this는 함수를 감싸고 있는 외부 스코프의 this를 참조한다 가 결론이다.

여기서, 화살표 함수가 만약 특정 객체의 프로퍼티로 할당이 되어 있다면 this의 참조는 어떻게 변할까?

생성자 함수?

화살표 함수의 경우, 기존 함수의 역할 중 생성자로서의 함수 의 역할을 수행하지 못한다. 즉, 객체를 생성하는 함수의 역할을 수행하지 못한다는 이야기이다. 다시 말해, new 키워드를 사용하여 객체를 생성하지 못한다.

하지만, 모든 함수가 Function 생성자의 인스턴스라는 점은 변하지 않는다.

객체 리터럴 강화

객체 리터럴 표현 사용 시, 기존 표현보다 좀 더 개선된 표현으로 코드 작성이 가능하다. 기존 객체 리터럴 표현식의 경우 키:벨류 형식으로 표현하는데 ES6에서는 리터럴에 추가할 프로퍼티명이 변수명과 동일하다면 그대로 아래와 같이 표현이 가능하다.

조합 프로퍼티 사용

해체 할당 포스팅에서 객체 프로퍼티를 동적으로 부여 할 수 있고, 이를 런타임 시 계산된 프로퍼티 값으로 쓸 수 있다고 했는데, 객체 리터럴 표현 역시 이런 것이 가능하다.

요약

  • 화살표 함수의 this 바인딩에 주의하자.
    • 화살표 함수를 감싸고 있는 외부 스코프의 this를 참조한다.
  • 화살표 함수는 인자의 갯수에 따라 소괄호 생략이 가능하며, 중괄호 코드 부분도 라인의 수에 따라 생략이 가능하다.
  • 객체 프로퍼티는 동적으로 부여 가능하며, 프로퍼티가 매소드인 경우 키 값을 생략한 표현이 가능하다.
